# Discovery questions

These questions will be refined through conversation with Paul and examples from the team.

## Product and users

1. Is the first release a public skill, an internal Chrome skill, or a public core with private overlays?
2. Who is the primary first-run user: DevRel, technical writer, PM, engineer, developer marketer, partnerships, or leadership?
3. Should one skill support feature, initiative, deprecation, adoption, support, and event modes, or should a router delegate to focused skills?
4. What evidence is mandatory before DevRel agrees to support a feature?
5. What should the skill do when evidence is weak: stop, create a research plan, or produce explicitly provisional artifacts?

## Canonical process

6. Which existing Chrome templates, launch reviews, GTM processes, planning docs, and aliases are authoritative?
7. Which parts can be encoded publicly, and which must remain private references or overlays?
8. Where should generated artifacts live, and what review/approval process applies?
9. Which process stages should trigger proactive DevRel involvement rather than waiting for a request?
10. Which roles own each stage and artifact?

## Outputs

11. Which artifacts should the first version generate directly?
12. What constitutes a complete “DevRel in a box” pack?
13. Should decks target Google Slides through an integration, portable Markdown/HTML, or both?
14. What demo quality and validation bar should apply? Can Chrome Platform Showcase supply the default demo workflow?
15. What forms should FAQs and support knowledge take so they improve canonical documentation rather than becoming a stale duplicate?

## Scale and measurement

16. What adoption outcomes matter by feature type?
17. Which leading indicators matter before broad usage data exists?
18. How should the skill assess partner quality and commitment?
19. How should it identify ecosystem multipliers such as frameworks, libraries, tools, GDEs, meetups, vendors, and education channels?
20. What existing launches represent excellent and poor outcomes, and why?

## Team operation

21. How are the approximately 35 team members organized by role, domain, region, and decision rights?
22. Which work must remain expert-led, and which repetitive work is safe to automate?
23. Where does work currently stall or get duplicated?
24. How much should the skill enforce a common quality bar versus adapting to domain-specific practice?
25. What feedback loop should update the skill as the team learns?
